# Prompt สำหรับ Cursor — ตรวจสอบ แก้ไข และพัฒนาเว็บไซต์ต่อ

> วิธีใช้: คัดลอกข้อความทั้งหมดด้านล่างนี้ไปวางในแชทของ Cursor (แนะนำใช้ Agent mode เพื่อให้ Cursor
> อ่าน/แก้ไขไฟล์ในโปรเจกต์ได้โดยตรง) แล้วปรับแก้ส่วนที่เป็น `[ ]` ให้ตรงกับข้อมูลจริงของคุณก่อนส่ง

## บริบทโปรเจกต์

นี่คือเว็บไซต์ Static (HTML5 + CSS3 + Vanilla JavaScript, ไม่มี framework, ไม่มีระบบหลังบ้าน)
สำหรับเว็บไซต์ส่วนตัว/ผู้สร้างคอนเทนต์ของ Boorapatis Ploysuwan โฮสต์อยู่บน GitHub ที่
`https://github.com/burapatis/website-2`

โครงสร้างไฟล์ปัจจุบัน:
```
.
├── index.html              หน้าหลัก
├── privacy.html             นโยบายความเป็นส่วนตัว
├── terms.html               ข้อตกลงการใช้งาน
├── content-policy.html      นโยบายเนื้อหา
├── styles.css               สไตล์ทั้งหมด (ใช้ร่วมกันทุกหน้า)
├── script.js                 interactive behavior (เมนูมือถือ, scroll reveal, back-to-top, ใช้ร่วมกันทุกหน้า)
├── robots.txt
├── sitemap.xml
├── README.md
└── assets/favicon.svg
```

โดเมนที่ใช้งานจริงของเว็บไซต์นี้คือ: `[ใส่โดเมนจริง เช่น https://burapatis.github.io/website-2/ หากยังไม่ได้ตั้ง custom domain]`
(หากยังไม่ตั้ง custom domain ค่าเริ่มต้นของ GitHub Pages สำหรับ repo นี้คือรูปแบบ
`https://<github-username>.github.io/<repo-name>/` — กรุณาตรวจสอบใน Settings → Pages ของ repo
เพื่อยืนยัน URL ที่ถูกต้องก่อนแก้ไขไฟล์)

## เป้าหมายของงานนี้

ช่วยตรวจสอบ แก้ไข เพิ่มเติม และพัฒนาเว็บไซต์นี้ต่อ ให้ **ถูกต้อง ครบถ้วน สอดคล้องกันทุกไฟล์**
ใช้งานได้จริงตามมาตรฐานเว็บ และมีความครอบคลุมทางกฎหมาย/นโยบายที่เหมาะสมยิ่งขึ้น โดยไม่เปลี่ยนแนวทาง
การออกแบบหรือคอนเซปต์เดิมของเว็บไซต์ (โทนสี navy/indigo/gold/teal/terracotta, ฟอนต์ Noto Serif Thai
+ Noto Sans Thai, ไม่ใช้ framework หนัก) เว้นแต่จะมีเหตุผลทางเทคนิคที่ชัดเจนและแจ้งให้ทราบก่อน

## รายการงานที่ต้องทำ (ทำตามลำดับ และสรุปผลทุกข้อในรายงานสุดท้าย)

### 1. ตรวจสอบความถูกต้องทางเทคนิค (Code QA)
- ตรวจสอบ HTML ทุกไฟล์ว่า tag เปิด/ปิดครบ, semantic ถูกต้อง, ไม่มี attribute ซ้ำซ้อนหรือผิดพลาด
- ตรวจสอบ CSS ว่า selector ไม่ชนกันจนยกเลิกกันเอง (specificity conflict), ไม่มี class ที่ประกาศแล้วไม่ได้ใช้
- ตรวจสอบ JavaScript ด้วย `node -c script.js` หรือเทียบเท่า ว่าไม่มี syntax error และไม่มี console error
  เมื่อเปิดเว็บไซต์จริง
- รันตรวจสอบลิงก์ภายในทั้งหมด (internal links/anchors) ว่าไม่มีลิงก์เสีย และ id ที่ถูกอ้างอิงด้วย `#anchor`
  มีอยู่จริงในหน้าปลายทาง
- ตรวจสอบว่าทุกหน้า (index, privacy, terms, content-policy) ใช้ header/footer/nav ที่ **เหมือนกันทุกตัวอักษร**
  (ยกเว้นส่วนที่ต้องต่างกัน เช่น title, active state) เพื่อความสอดคล้องกัน — ถ้าพบความต่างที่ไม่ตั้งใจ ให้แก้ไขให้ตรงกัน

### 2. แก้ไขโดเมนและ SEO metadata
- แทนที่ `example.com` ทั้งหมดในทุกไฟล์ (`index.html`, `privacy.html`, `terms.html`, `content-policy.html`,
  `robots.txt`, `sitemap.xml`) ด้วยโดเมนจริงที่ระบุไว้ด้านบน
- ตรวจสอบ JSON-LD ใน `index.html` ว่า syntax ถูกต้อง (validate เป็น JSON ได้จริง) และ `sameAs` ลิงก์ไปยัง
  ช่องทางที่ถูกต้องครบทุกช่อง
- ตรวจสอบว่า meta title/description ของแต่ละหน้าไม่ซ้ำกัน และมีความยาวเหมาะสมกับ SEO
  (title ~50-60 ตัวอักษร, description ~150-160 ตัวอักษร)
- หากยังไม่มีภาพ `assets/og-cover.jpg` ให้สร้างไฟล์ placeholder หรือแจ้งเตือนว่าต้องเพิ่มภาพจริงก่อน deploy
  (ห้ามใช้ภาพที่มีปัญหาลิขสิทธิ์)

### 3. Accessibility audit
- ตรวจสอบ contrast ratio ของสีตัวอักษรกับพื้นหลังทุกคู่สีหลักว่าผ่านมาตรฐาน WCAG AA เป็นอย่างน้อย
- ตรวจสอบว่าทุก interactive element (ปุ่ม, ลิงก์, เมนูมือถือ) มี focus state ที่มองเห็นได้ และ aria-label/aria-expanded
  ถูกต้องครบถ้วน
- ตรวจสอบ heading hierarchy (h1 → h2 → h3) ในทุกหน้าว่าไม่ข้ามระดับ
- ตรวจสอบว่า `prefers-reduced-motion` ถูก respect จริงในทุก animation (scroll reveal, signal divider)

### 4. Responsive & cross-browser
- ทดสอบ layout ที่ breakpoint หลัก: มือถือ (~375px), แท็บเล็ต (~768px), เดสก์ท็อป (~1280px+)
- ตรวจสอบเมนูมือถือ (hamburger menu) ว่าเปิด/ปิดได้ปกติ และปิดอัตโนมัติเมื่อคลิกลิงก์
- แจ้งหากพบปัญหาการแสดงผลที่ต่างกันใน Chrome/Safari/Firefox ที่ควรแก้ไข

### 5. ตรวจสอบความสอดคล้องด้านกฎหมาย/นโยบาย (สำคัญที่สุด)
- เปรียบเทียบเนื้อหาใน `privacy.html`, `terms.html`, `content-policy.html` กับการทำงานจริงของเว็บไซต์ในโค้ด
  เช่น:
  - ถ้า `privacy.html` ระบุว่า **"ไม่มีการใช้คุกกี้ของเว็บไซต์เอง"** ให้ตรวจสอบโค้ดทั้งหมดว่าไม่มีการตั้งค่า
    cookie, localStorage, หรือ tracking script ใด ๆ ที่ขัดกับคำกล่าวนี้ (ปัจจุบันใช้ Google Fonts เท่านั้น)
  - ถ้ามีการเพิ่มเครื่องมือใหม่ในอนาคต (เช่น Google Analytics, Facebook Pixel, ระบบฟอร์ม, ระบบ comment)
    **ต้องแจ้งเตือนผู้ใช้ทันทีว่าต้องอัปเดต `privacy.html` ให้ตรงกับการเก็บข้อมูลจริง ก่อน deploy** — ห้ามเพิ่ม
    เครื่องมือเหล่านี้แบบเงียบ ๆ โดยไม่แก้ไขนโยบายให้ตรงกัน
  - ตรวจสอบว่าอีเมลติดต่อ (`burapatis@gmail.com`) ตรงกันทุกที่ที่ปรากฏ (index, privacy, terms, content-policy, footer)
- ตรวจสอบว่า URL ของช่อง YouTube/TikTok ที่ปรากฏใน `index.html`, footer ทุกหน้า, และ JSON-LD ตรงกันทั้งหมด
  และยังเป็น URL ที่ถูกต้อง/ใช้งานได้จริง
- ตรวจสอบวันที่ "ปรับปรุงล่าสุด" ใน 3 หน้านโยบาย — หากมีการแก้ไขเนื้อหาในหน้าใด ให้อัปเดตวันที่นั้นด้วย
- **ห้าม Cursor เขียนหรือแก้ไขข้อความในหน้านโยบายให้สื่อว่าเว็บไซต์รับประกันความถูกต้องทางกฎหมาย 100%**
  หรือลบข้อความ disclaimer ที่ระบุว่า "ไม่ใช่คำแนะนำทางกฎหมาย" ออกโดยไม่ได้รับการยืนยันจากเจ้าของเว็บไซต์ก่อน
- หากพบส่วนใดที่ต้องการการตัดสินใจทางกฎหมายเฉพาะทาง (เช่น การปฏิบัติตาม PDPA แบบเต็มรูปแบบสำหรับกรณีใช้งาน
  เชิงพาณิชย์) ให้ระบุไว้เป็นรายการแยกในรายงานสรุปว่า **"ควรปรึกษาผู้เชี่ยวชาญด้านกฎหมายเพิ่มเติม"** แทนการแก้ไข
  หรือสมมติเองโดยไม่มีข้อมูลยืนยัน

### 6. งานเสริมที่ควรทำให้สมบูรณ์ขึ้น
- เพิ่ม favicon หลายขนาด (เช่น 16×16, 32×32, 180×180 สำหรับ apple-touch-icon, 192×192/512×512 สำหรับ
  Android) และพิจารณาเพิ่มไฟล์ `site.webmanifest` พื้นฐาน
- ตรวจสอบและจัดรูปแบบ commit message ให้สื่อความหมายชัดเจนสำหรับการแก้ไขแต่ละชุด (อย่า commit รวมทุกอย่าง
  เป็น 1 commit เดียว ให้แยกตามหมวดงานข้างบน)
- หากมี GitHub Actions หรือ CI ที่เกี่ยวข้องกับการ deploy GitHub Pages ให้ตรวจสอบว่าทำงานถูกต้อง

## ข้อจำกัดที่ต้องรักษาไว้ (Constraints)

1. ห้ามเปลี่ยนไปใช้ framework (React/Vue/ฯลฯ) หรือ build tool ที่ทำให้ deploy บน GitHub Pages ยุ่งยากขึ้น
   เว้นแต่จะถูกขอให้ทำโดยเฉพาะ
2. ห้ามเพิ่ม dependency หรือ library ภายนอกที่ไม่จำเป็น
3. รักษาภาษาไทยเป็นภาษาหลักของเนื้อหา และคงโทนภาษาสุภาพ ชัดเจน มีวุฒิภาวะตามที่มีอยู่เดิม
4. ห้ามใช้รูปภาพจากภายนอกที่ไม่ทราบสิทธิ์การใช้งาน — หากต้องเพิ่มภาพ ให้ใช้ placeholder (gradient/SVG)
   เหมือนที่ใช้อยู่เดิม หรือแจ้งให้เจ้าของเว็บไซต์เตรียมภาพที่มีสิทธิ์ใช้งานถูกต้องเอง
5. ห้ามลบหรือเปลี่ยนแปลงเนื้อหาหลักการ/นโยบาย/ข้อมูลติดต่อ โดยไม่มีเหตุผลทางเทคนิคหรือความสอดคล้องที่ชัดเจน

## สิ่งที่ต้องส่งกลับมา (Deliverable)

หลังตรวจสอบและแก้ไขเสร็จ กรุณาสรุปเป็นรายงานท้ายงาน ครอบคลุม:
1. รายการปัญหาที่พบและแก้ไขแล้ว (แยกตามหมวดข้อ 1–6 ข้างบน)
2. รายการที่ **ยังไม่ได้แก้ไข** เพราะต้องให้เจ้าของเว็บไซต์ตัดสินใจเอง (เช่น เนื้อหาที่ต้องเลือกด้วยตัวเอง,
   ประเด็นที่ต้องปรึกษาผู้เชี่ยวชาญด้านกฎหมาย, ภาพ/โลโก้จริงที่ต้องเตรียมเอง)
3. คำแนะนำขั้นต่อไปก่อน deploy เว็บไซต์ขึ้นใช้งานจริงแบบสาธารณะ
